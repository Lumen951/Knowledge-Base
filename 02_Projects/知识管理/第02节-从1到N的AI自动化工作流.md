## 二、知识系统设计：用 PARA 驯服信息洪流

### 1. 为什么需要组织方法？

#### 回顾：我们在第一节建立的文件夹

```
ai-knowledge-base/
├── inbox/        # 收件箱
├── notes/        # 笔记
└── projects/     # 项目
```

**这个结构的问题：**
- 太粗糙：只有 3 个分类，无法应对复杂场景
- 边界模糊：什么算"笔记"？什么算"项目"？
- 缺少动态性：信息的状态会变化，但文件夹是静态的

**真实场景：**

当你开始积累知识时，你会发现：
- 有些内容是**长期关注的领域**（如 AI 学习、产品运营）
- 有些内容是**有明确期限的项目**（如毕业论文、创业计划）
- 有些内容是**参考资料**（如教程、论文 PDF、网页摘录）
- 有些内容是**未来可能用到的**（如想法、灵感、待办事项）

**你需要一套能够动态分类、清晰定位的组织方法。**

---

### 2. PARA 方法：为信息找到归属

PARA 是 Tiago Forte 提出的知识管理框架，代表四个层级：

| 层级    | 英文        | 中文  | 定义            | 时间属性      | 例子             |
| ----- | --------- | --- | ------------- | --------- | -------------- |
| **P** | Projects  | 项目  | 有明确目标和截止日期的任务 | 短期（数周-数月） | 毕业论文、创业BP、课程作业 |
| **A** | Areas     | 领域  | 长期关注的责任领域     | 长期（持续维护）  | AI学习、产品运营、职业规划 |
| **R** | Resources | 资源  | 可能有用的参考材料     | 无时限       | 教程合集、论文库、工具清单  |
| **A** | Archives  | 归档  | 已完成或不再活跃的内容   | 已结束       | 已完成的项目、过期的笔记   |

#### PARA 的核心逻辑

```
按"可操作性"降序排列：
Projects（最可操作）
    ↓
Areas（持续行动）
    ↓
Resources（备查）
    ↓
Archives（不再活跃）
```

**关键原则：**
- **Project 优先**：任何与当前项目相关的内容，优先放入 Projects
- **Area 是容器**：长期关注的主题，用 Areas 承载
- **Resource 是素材库**：不确定何时用的，先放 Resources
- **Archive 是归宿**：完成的项目、过期的内容，定期归档

### 5. 用 Dataview 让 PARA 动起来

#### 什么是 Dataview？

Dataview 是 Obsidian 最强大的插件之一，它可以：
- 把你的笔记当作数据库查询
- 自动生成列表、表格、统计数据
- 根据标签、文件夹、frontmatter 动态聚合内容

#### 在 PARA Index 中使用 Dataview

创建文件：`01_Projects/Index.md`

````markdown
# 📋 当前活跃项目

## 高优先级项目（自动）

```dataview
TABLE
  deadline as "截止日期",
  status as "状态"
FROM "01_Projects"
WHERE type = "project" AND priority = "A"
SORT deadline ASC
```

## 本周待办（自动）

```dataview
TASK
FROM "01_Projects"
WHERE status = "active"
```
````

**效果：**
- 自动列出所有高优先级项目
- 按截止日期排序
- 实时显示待办事项

## 三、Git 进阶：从"会用"到"理解"

### 1. 回顾：第一节学到的 Git 操作

---

### 2. Git 的核心概念：三个区域

```
工作区（Working Directory）  →  暂存区（Staging Area）  →  本地仓库（Local Repository）  →  远程仓库（Remote）
     ↓                              ↓                            ↓                              ↓
  你正在编辑的文件               git add 后的文件          git commit 后的快照         git push 后的云端备份
```

| 区域 | 说明 | 常用命令 |
|------|------|---------|
| **工作区** | 你看到的文件夹 | `ls`, `cat` |
| **暂存区** | 准备提交的文件 | `git add` |
| **本地仓库** | 提交历史（快照） | `git commit` |
| **远程仓库** | GitHub/Gitee 等 | `git push`, `git pull` |

#### 通俗理解

**工作区 = 你的办公桌**
- 摊开的文件、草稿、修改中的内容

**暂存区 = 打包箱**
- 你挑选出"确定要寄出"的文件，放进箱子

**本地仓库 = 快递单据**
- 你贴上标签（commit message），确认打包

**远程仓库 = 云端仓库**
- 你把箱子寄到云端，其他人也能取

---

### 3. 日常工作流：pull → edit → add → commit → push

#### 完整流程（每天开始工作前）

```bash
# 第1步：拉取最新内容（如果多设备同步）
git pull origin main

# 第2步：开始编辑你的笔记
# 在 Obsidian 中修改、添加笔记...

# 第3步：查看当前状态
git status

# 第4步：添加到暂存区
git add .                  # 添加所有修改
# 或者
git add 01_Projects/      # 只添加特定文件夹

# 第5步：提交到本地仓库
git commit -m "feat: 完成毕业论文文献综述初稿"

# 第6步：推送到远程仓库
git push origin main
```

#### 提交信息（Commit Message）规范

**格式：**
```
<type>: <subject>

<body>（可选）
```

**常用 type：**
| Type | 说明 | 例子 |
|------|------|------|
| `feat` | 新增功能/内容 | `feat: 添加AI工具评测笔记` |
| `fix` | 修复错误 | `fix: 更正论文引用格式` |
| `docs` | 文档更新 | `docs: 完善PARA使用指南` |
| `refactor` | 重构/重新组织 | `refactor: 重构知识库结构` |
| `chore` | 日常维护 | `chore: 清理Inbox文件夹` |

**示例：**
```bash
git commit -m "feat: 完成第二节课笔记整理

- 添加PARA组织方法说明
- 创建项目主控模板
- 整理Git工作流笔记"
```

### 5. 分支管理：安全地试错

#### 为什么需要分支？

**场景：**
你想尝试一个新的笔记组织方法，但不确定是否有效。如果直接在主分支修改，可能会把现有内容搞乱。

**解决方案：创建实验分支**

```bash
# 创建并切换到新分支
git checkout -b experiment-new-structure

# 在这个分支上随意试验...
# 修改文件、重新组织...

# 如果满意，合并回主分支
git checkout main
git merge experiment-new-structure

# 如果不满意，直接删除分支
git branch -d experiment-new-structure
```

#### 分支使用场景

| 场景 | 分支名 | 说明 |
|------|--------|------|
| **日常工作** | `main` | 主分支，稳定版本 |
| **新功能实验** | `feature/new-template` | 测试新模板系统 |
| **大规模重构** | `refactor/para-migration` | 迁移到PARA结构 |
| **临时想法** | `draft/idea-20251124` | 临时草稿，随时删除 |

#### 课堂练习（10分钟）

**任务：用分支尝试重命名文件夹**

```bash
# 1. 创建实验分支
git checkout -b experiment-rename

# 2. 重命名文件夹（在Obsidian或文件管理器中）
# 比如把 "notes" 改为 "02_Areas"

# 3. 提交修改
git add .
git commit -m "experiment: 尝试重命名为PARA结构"

# 4. 切换回主分支（文件会恢复原样）
git checkout main

# 5. 如果满意实验结果，合并分支
git merge experiment-rename

# 6. 删除实验分支
git branch -d experiment-rename
```

### 6. 常见问题与解决

#### 问题1：提交后发现写错了，怎么撤销？

**场景：**刚刚 commit 了一个文件，但发现里面有错别字

**解决：**
```bash
# 修改文件后，追加到上一次提交
git add .
git commit --amend --no-edit

# 或者修改提交信息
git commit --amend -m "feat: 修正后的提交信息"
```

#### 问题2：不小心 add 了不该提交的文件

**场景：**`git add .` 后发现包含了临时文件

**解决：**
```bash
# 取消暂存特定文件
git reset HEAD 临时文件.tmp

# 或取消所有暂存
git reset HEAD .
```

#### 问题3：多设备同步冲突

**场景：**在公司电脑修改了笔记，回家后在家里电脑也修改了同一文件，push 时冲突

**解决：**
```bash
# 先拉取远程更新
git pull origin main

# Git会提示冲突，打开冲突文件
# 手动解决冲突（删除<<<<<<< ======= >>>>>>>标记）

# 解决后提交
git add .
git commit -m "fix: 解决多设备同步冲突"
git push origin main
```

### 2. 教学重点与难点

#### 重点1：PARA 的动态理解

**易错点：**学生容易把 PARA 当作"四个文件夹"
**强调：**PARA 是"可操作性降序"的组织逻辑，而非静态分类

**演示建议：**
- 用真实案例展示"一个笔记如何从 Inbox → Projects → Archives 流动"
- 强调"每周清理 Inbox"的重要性

#### 重点2：拖拽文件技巧的演示

**易错点：**学生不知道可以直接拖拽文件到终端
**强调：**拖拽是最简单的路径获取方式，不需要手动输入路径

**演示建议：**
- 现场演示：打开终端 → 输入提示词 → 拖拽文件 → 终端自动填充路径
- 演示拖拽多个文件的场景（按住Shift或Ctrl选择多个）
- 强调"暂停，不要按回车，先拖拽文件"

#### 重点3：Git 工作流的日常化

**易错点：**学生会忘记 `git pull`，导致冲突
**强调：**每天开始工作前先 pull，结束后 commit + push（使用GitHub Desktop更简单）

**演示建议：**
- 模拟多设备同步场景
- 故意制造冲突，演示如何解决

### 3. 常见问题预案

**Q1：PARA 和标签（tags）有什么区别？**
- **A1**：PARA 是物理位置（文件夹），tags 是逻辑标签。PARA 用于组织，tags 用于检索。

**Q2：每次修改都要 commit 吗？**
- **A2**：不是。建议"完成一个逻辑单元"就 commit（如"完成一篇笔记"或"整理完一个项目"）。

**Q3：拖拽文件到终端没反应？**
- **A3**：确保光标在终端输入行，有些终端可能需要按住Shift拖拽。VSCode的集成终端完全支持拖拽。

**Q4：如何处理剪贴板内容？**
- **A4**：大部分AI CLI工具可以直接读取剪贴板，或者使用 `--clipboard` 选项。不同工具可能略有差异。
